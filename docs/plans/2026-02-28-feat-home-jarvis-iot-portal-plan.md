# feat: Build Home Jarvis IoT Integration Portal

## Overview

A locally-hosted web portal that unifies 8 smart home device brands under one admin UI and bridges them into Apple HomeKit via a HAP bridge. The server runs on Bun + Elysia + TypeScript. The frontend is React + Tailwind + react-aria. Everything lives on the home network — Apple HomeKit handles remote access via HomePod/Apple TV.

**Key constraint:** No auth, no cloud hosting, no Tailscale for v1. This is a trusted local network admin tool.

*(see brainstorm: docs/brainstorms/2026-02-28-home-jarvis-brainstorm.md)*

---

## Problem Statement

Smart home devices span 8+ brands with incompatible apps, APIs, and discovery mechanisms. No single hub provides:
- Unified visibility across all brands
- Local-first control and privacy
- A clean bridge into Apple HomeKit without paying for Homebridge cloud or running a heavy platform

Home Jarvis solves this as a lightweight, self-hosted TypeScript application.

## Database Schema

### `integrations`
Stores per-brand credentials and enabled state.

```typescript
// server/src/db/schema.ts
const integrations = sqliteTable('integrations', {
  id: text('id').primaryKey(),                        // uuid
  brand: text('brand').notNull().unique(),            // 'hue' | 'govee' | 'vesync' | ...
  config: text('config').notNull(),                   // JSON: { apiKey, email, password, token, ... }
  enabled: integer('enabled', { mode: 'boolean' }).notNull().default(true),
  createdAt: integer('created_at').notNull(),
  updatedAt: integer('updated_at').notNull(),
})
```

### `devices`
All discovered devices, one row per device.

```typescript
const devices = sqliteTable('devices', {
  id: text('id').primaryKey(),                        // uuid (portal-assigned, stable)
  integrationId: text('integration_id').references(() => integrations.id),
  brand: text('brand').notNull(),
  externalId: text('external_id').notNull(),          // brand-native device ID
  name: text('name').notNull(),
  type: text('type').notNull(),                       // 'light' | 'switch' | 'thermostat' | 'air_purifier' | 'sensor'
  state: text('state').notNull().default('{}'),       // JSON: { on, brightness, temperature, ... }
  online: integer('online', { mode: 'boolean' }).notNull().default(true),
  homekitEnabled: integer('homekit_enabled', { mode: 'boolean' }).notNull().default(false),
  homekitUuid: text('homekit_uuid'),                  // stable UUID used by hap-nodejs
  lastSeen: integer('last_seen'),
  createdAt: integer('created_at').notNull(),
  updatedAt: integer('updated_at').notNull(),
}, (t) => ({
  uniq: unique().on(t.brand, t.externalId),           // deduplicate by (brand, externalId)
}))
```

### `homekit_config`
Singleton row for HAP bridge state.

```typescript
const homekitConfig = sqliteTable('homekit_config', {
  id: text('id').primaryKey().default('singleton'),
  pin: text('pin').notNull(),                         // formatted as '123-45-678'
  username: text('username').notNull(),               // MAC-like string for HAP
  port: integer('port').notNull().default(51826),
  paired: integer('paired', { mode: 'boolean' }).notNull().default(false),
  createdAt: integer('created_at').notNull(),
})
```

## Adapter Interface

Every integration implements this contract. Adding a new brand means writing one adapter file.

```typescript
// server/src/integrations/types.ts

export interface DiscoveredDevice {
  externalId: string       // brand-native ID
  name: string
  type: DeviceType         // 'light' | 'switch' | 'thermostat' | 'air_purifier' | 'sensor'
  state: DeviceState
  online: boolean
}

export interface DeviceState {
  on?: boolean
  brightness?: number      // 0–100
  colorTemp?: number       // Kelvin
  color?: { r: number; g: number; b: number }
  temperature?: number     // Celsius
  humidity?: number
  fanSpeed?: number        // 0–100
  airQuality?: number
  [key: string]: unknown   // brand-specific extras
}

export interface DeviceAdapter {
  readonly brand: string
  readonly displayName: string
  readonly discoveryMethod: 'local' | 'cloud' | 'both'

  // Validate credentials and throw if invalid
  validateCredentials(config: Record<string, string>): Promise<void>

  // Return all devices this integration knows about
  discover(): Promise<DiscoveredDevice[]>

  // Get current device state
  getState(externalId: string): Promise<DeviceState>

  // Apply state change
  setState(externalId: string, state: Partial<DeviceState>): Promise<void>

  // Map to HAP Accessory. Return null for adapters that should not bridge (Aqara).
  toHomeKitAccessory(device: Device): HAPAccessory | null
}
```

## HAP Bridge Architecture

The HAP bridge runs **in the same Bun process** as Elysia, on a separate TCP port (default: 51826). mDNS (UDP 5353) for bridge advertisement coexists cleanly with the HTTP server.

```typescript
// server/src/homekit/bridge.ts (concept)

import { Bridge, Categories } from 'hap-nodejs'
import { uuid } from 'hap-nodejs'

const bridge = new Bridge('Home Jarvis', uuid.generate('home-jarvis-bridge'))

bridge.publish({
  username: homekitConfig.username,     // from DB — stable across restarts
  pincode: homekitConfig.pin,
  port: homekitConfig.port,
  category: Categories.BRIDGE,
})

// After publish, generate QR code URI
const setupUri = bridge.setupURI()      // convert to QR code image in UI
```

**Dynamic accessory management (no re-pairing required):**
```typescript
// Expose a device:
bridge.addBridgedAccessory(accessory)

// Unexpose a device:
bridge.removeBridgedAccessory(accessory)
// hap-nodejs calls updateAdvertisement() internally — HomeKit notified automatically
```

**UUID stability:** All accessory UUIDs are generated from a stable seed (device `id` from DB) using `uuid.generate(device.id)`. Server restarts produce identical UUIDs.

## Integration-Specific Notes

### Philips Hue
- **Discovery:** N-UPnP via `https://discovery.meethue.com` (< 500ms, preferred) + mDNS fallback
- **API:** CLIP API v2, local REST over HTTPS to bridge IP
- **Auth:** Press button on physical bridge once to get API token (stored in `config`)
- **Real-time:** Hue supports SSE events from the bridge — subscribe for instant state updates instead of polling
- **Library:** `node-hue-api` or direct fetch to `https://<bridge-ip>/api/<token>/lights`

### Govee
- **Discovery:** Cloud API only (no local discovery)
- **API:** `https://developer-api.govee.com/v1` — requires API key from Govee developer portal
- **Rate limit:** 10,000 requests per 24 hours + 10 state changes per minute
- **Polling strategy:** Device list: every 5 minutes. State: every 60 seconds minimum. Respect `X-RateLimit-Remaining` headers.
- **MQTT:** Govee also offers MQTT for real-time updates — optional enhancement for v2

### VeSync (Levoit Air Purifiers)
- **Discovery:** Cloud API only
- **API:** `https://smartapi.vesync.com` — unofficial, reverse-engineered. Login with email/password to get `accountId` + `authToken`
- **Rate limit:** Cache device state reads for minimum 5 seconds. Add 500ms delay between successive API calls.
- **Auth:** No public OAuth — store email/password, exchange for token on startup, handle token refresh
- **Library reference:** `node-vesync` npm package or implement from `pyvesync` docs

### LG ThinQ
- **⚠️ CRITICAL:** LG uses **OAuth 2.0**, not a static API key or username/password. This requires a different credential form: an "Authorize with LG" button that opens LG's OAuth page, redirects back with an auth code, then exchanges for access/refresh tokens. Token refresh logic is required.
- **API:** ThinQ v2 REST API
- **Impact:** The integration setup page needs to support an OAuth redirect flow for LG. Store `accessToken`, `refreshToken`, `expiresAt`.

### GE (Cync / SmartHQ)
- **Discovery:** Cloud API
- **API:** SmartHQ REST API — requires GE account credentials
- **Note:** GE's API is not publicly documented; implementation should reference Home Assistant's GE integration as a reference

### Aqara
- **Discovery:** Local mDNS + cloud API
- **HomeKit:** **Do NOT bridge** — Aqara devices are natively HomeKit compatible. Show in dashboard for unified visibility only.
- **UI rule:** HomeKit toggle is **disabled** with tooltip: "Aqara supports HomeKit natively. Add via the Apple Home app."

### SmartThings
- **Discovery:** Cloud API
- **API:** SmartThings REST API with personal access token (PAT)
- **Note:** v1 — user is still setting up SmartThings. Include integration, but may have limited device coverage initially.

### Resideo (Honeywell Home)
- **Discovery:** Cloud API
- **API:** Resideo Developer API — requires developer account and API key
- **Devices:** Thermostats, leak sensors, smoke detectors

## Core Portal Features

### 1. Integration Setup (`/integrations`)

Each brand has a credential form. Most brands use key/value fields. LG uses an OAuth button.

```
POST /api/integrations
Body: { brand: 'govee', config: { apiKey: '...' } }

Server: validates credentials via adapter.validateCredentials()
        saves to DB on success
        triggers initial discovery for this integration
```

**Integration removal flow:**
1. Check if any devices from this integration are HomeKit-exposed
2. If yes: warn user and require confirmation
3. On confirm: unexpose all devices, remove from HAP bridge, delete devices, delete integration

### 2. Device Discovery Engine (`server/src/discovery/`)

Two parallel systems:

**mDNS Scanner** (`mdns-scanner.ts`)
- Runs on startup
- Uses `@homebridge/ciao` (already a hap-nodejs dependency, handles Bonjour correctly)
- Looks for `_hue._tcp.local`, `_aqara._tcp.local`, etc.
- On discovery: upsert to `devices` table via `(brand, externalId)` unique constraint

**Cloud Poller** (`cloud-poller.ts`)
- Runs per-integration on configurable intervals (integration-specific defaults)
- Govee: 60-second state poll, 5-minute device list poll
- Others: 30-second state poll (or use push if available)
- On each poll: calls `adapter.discover()`, upserts to DB, marks missing devices as `online = false`

**Device disappearance behavior:**
- If a device is not seen for 2 poll cycles: mark `online = false` in DB
- If `online = false` and device was HomeKit-exposed: device goes "not responding" in Apple Home (expected behavior — HomeKit handles this gracefully)
- Never auto-delete a device — user must manually remove

### 3. Device Dashboard (`/` on frontend)

Live SSE stream delivers device state to the React frontend.

```typescript
// server/src/routes/events.controller.ts
app.get('/api/events', ({ set }) => {
  return new Response(
    new ReadableStream({
      start(controller) {
        // emit current state immediately on connect
        // subscribe to internal event bus for state changes
        // re-emit on every device update
      }
    }),
    { headers: { 'Content-Type': 'text/event-stream', 'Cache-Control': 'no-cache' } }
  )
})
```

Frontend SSE hook reconnects automatically with exponential backoff. Stale connection shows a "Reconnecting..." indicator.

**Dashboard layout:**
- Group devices by brand (initial v1 — no custom rooms)
- Each `DeviceCard` shows: name, brand, type icon, online status, current state, HomeKit toggle
- Empty state: shows CTA to `/integrations` if no integrations configured

### 4. HomeKit Toggle

```
PATCH /api/devices/:id/homekit
Body: { enabled: true | false }
```

Server logic:
1. Fetch device from DB
2. If `enabled = true`:
   - If device is Aqara: return 400 "Aqara supports HomeKit natively"
   - Call `accessory-factory.ts` to create HAP accessory
   - `bridge.addBridgedAccessory(accessory)`
   - Save `homekitEnabled = true`, `homekitUuid` to DB
3. If `enabled = false`:
   - Look up HAP accessory by UUID
   - `bridge.removeBridgedAccessory(accessory)`
   - Save `homekitEnabled = false` to DB
4. Broadcast state change via SSE

### 5. HomeKit Pairing (`/homekit` on frontend)

On startup, the HAP bridge always starts (even if unpaired). The frontend shows:

- **If not paired:** Large QR code + PIN (`bridge.setupURI()` → rendered as QR image via `qrcode` package)
- **If paired:** "Paired ✓" indicator + "Reset pairing" button
- **Bridge health:** Status badge showing bridge running/stopped + last heartbeat

**Pairing page note:** If user toggles devices before pairing, they appear in the bridge but won't show in Apple Home until pairing is complete. This is expected behavior — no additional UI needed.

## Accessory Factory

Maps portal device types to HAP service types:

```typescript
// server/src/homekit/accessory-factory.ts

import { Accessory, Service, Characteristic, Categories, uuid } from 'hap-nodejs'

export function createHAPAccessory(device: Device, adapter: DeviceAdapter): HAPAccessory {
  const accessory = new Accessory(device.name, uuid.generate(device.id))

  accessory.getService(Service.AccessoryInformation)!
    .setCharacteristic(Characteristic.Manufacturer, device.brand)
    .setCharacteristic(Characteristic.Model, device.type)
    .setCharacteristic(Characteristic.SerialNumber, device.externalId)

  switch (device.type) {
    case 'light':
      const lightService = accessory.addService(Service.Lightbulb)
      lightService.getCharacteristic(Characteristic.On)
        .onGet(async () => (await adapter.getState(device.externalId)).on ?? false)
        .onSet(async (value) => adapter.setState(device.externalId, { on: value as boolean }))
      // Add Brightness if supported
      // Add ColorTemperature if supported
      break

    case 'switch':
      accessory.addService(Service.Switch)
        .getCharacteristic(Characteristic.On)
        .onGet(async () => (await adapter.getState(device.externalId)).on ?? false)
        .onSet(async (value) => adapter.setState(device.externalId, { on: value as boolean }))
      break

    case 'thermostat':
      accessory.addService(Service.Thermostat)
      // Map CurrentTemperature, TargetTemperature, CurrentHeatingCoolingState
      break

    case 'air_purifier':
      accessory.addService(Service.AirPurifier)
      // Map Active, CurrentAirPurifierState, TargetAirPurifierState
      break

    case 'sensor':
      accessory.addService(Service.TemperatureSensor)
      // or HumiditySensor, AirQualitySensor depending on device.state fields
      break
  }

  return accessory
}
```

## System-Wide Impact

### Interaction Graph

```
User toggles HomeKit in UI
  → PATCH /api/devices/:id/homekit
    → bridge.addBridgedAccessory() or removeBridgedAccessory()
      → hap-nodejs calls updateAdvertisement() via ciao mDNS
        → Apple Home app detects configuration change
          → HomePod/Apple TV updates its accessory registry
```

```
Cloud poller fires (every N seconds)
  → adapter.discover() + adapter.getState()
    → DB upsert (devices table)
      → SSE event bus emits state change
        → All connected browser clients receive update
          → React re-renders DeviceCard
```

```
HAP characteristic .onGet() fires (Apple Home polls state)
  → adapter.getState(device.externalId)
    → Cloud API call (or local Hue API call)
      → returns current state to HAP
        → Apple Home displays correct value
```

### Error & Failure Propagation

| Failure | Effect | Handling |
|---|---|---|
| Cloud API rate limit hit (Govee) | State becomes stale | Show "stale data" indicator on DeviceCard; log warning; retry after reset window |
| Cloud API credential expires (VeSync token, LG OAuth) | `discover()` and `getState()` throw | Surface per-integration error banner on Integrations page; trigger re-auth flow |
| HAP bridge crash | Apple Home devices go "not responding" | Status badge on HomeKit page goes red; auto-restart not implemented v1 |
| mDNS scan finds no Hue bridge | Dashboard empty for Hue | Show "No bridge found" on Integrations page for Hue; suggest checking IP |
| SQLite corruption | Server fails to start | Critical — backup strategy is v2 concern; document file location |
| SSE connection drops | Dashboard shows stale state | `useDeviceStream.ts` reconnects with exponential backoff; show "reconnecting" banner |

### State Lifecycle Risks

- **HomeKit accessory leak:** If server crashes after `addBridgedAccessory()` but before saving `homekitEnabled = true` to DB, the accessory exists in the bridge's in-memory state but not in DB. On restart, it won't be restored. Mitigation: write to DB first, then add to bridge.
- **Stale devices:** A device removed from the source integration (e.g., Govee bulb sold) stays in DB with `online = false`. If it was HomeKit-exposed, it stays as "not responding" in Apple Home indefinitely. Provide a manual "Remove device" action on the dashboard.
- **HAP pairing data vs DB:** hap-nodejs stores pairing state in `server/data/.homekit/`. If this folder is deleted but DB still shows `paired = true`, the bridge will re-advertise as unpaired but DB says otherwise. Mitigation: the `/api/homekit/reset` endpoint deletes both the file and the DB record together.

### Integration Test Scenarios

1. **Cold start with no integrations:** Server starts → dashboard shows empty state with CTA → add Govee integration → discovery runs → devices appear on dashboard.
2. **HomeKit toggle cycle:** Toggle device ON → verify appears in Apple Home → toggle OFF → verify disappears from Apple Home → server restart → device stays OFF, not re-exposed.
3. **Rate limit recovery:** Govee adapter hits 10,000/day limit → requests stop → state marked stale → next day (limit resets) → polling resumes automatically.
4. **Hue SSE vs poll:** Change a Hue light via physical switch → event arrives via Hue SSE subscription → dashboard updates within ~200ms without waiting for poll cycle.
5. **Integration credential rotation:** Update Govee API key → `validateCredentials()` passes → old key discarded → polling continues without interruption.

## Risk Analysis

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| LG ThinQ OAuth token rotation breaks polling | High | High | Implement token refresh 30min before expiry; surface re-auth prompt on 401 |
| Govee rate limit hit during demo/testing | High | Medium | Conservative polling defaults; rate limit guard enforced at adapter level |
| VeSync API changes (unofficial) | Medium | High | Reference pyvesync for latest endpoint patterns; VeSync adapter is isolated |
| GE Cync API is undocumented | High | Medium | Use Home Assistant SmartHQ integration as reference; may require significant reverse-engineering |
| hap-nodejs pairing breaks on server IP change | Low | High | Document that server must have static DHCP lease; note in setup instructions |
